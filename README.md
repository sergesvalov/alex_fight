# Alex Fight

> 3D-шутер | Godot 4.7 | Android / PC / VR (Quest 2) | Ретрофутуризм | P.T.-style Horror

Коридорный шутер с элементами survival horror и психологического триллера. Действие разворачивается в заброшенной провинциальной гостинице «Сибирь», которая превратилась в сюрреалистическую бесконечную петлю. Герой — полицейский без памяти (Алекс) — ищет выход на крышу, собирает голографические VHS-кассеты с отголосками своего прошлого и отбивается от жуткого проекта «Цербер».

---

## 🌟 Ключевые особенности

- **Бесконечная петля и Этажи (Non-Euclidean Geometry)**: 3 и 4 этажи гостиницы. Г-образная лестничная клетка работает как физический портал — бесшовная телепортация создаёт пугающий эффект лабиринта в духе P.T. Если вы попытаетесь спуститься, нижняя дверь вернет вас на тот же этаж.
- **Таинственные Двери-Порталы**: после сбора всех VHS-кассет этажа в стене комнаты прорезается таинственный выход — телепорт на другой этаж с перемешиванием кассет и монстра.
- **VR-Адаптация (Dual-Mode)**: Игра нативно поддерживает виртуальную реальность для Meta Quest 2 (OpenXR). Тот же исходный код работает и как классическая игра на телефоне/PC, и как полноценный VR-шутер с 6DoF, отслеживанием рук и парящим 3D-интерфейсом.
- **Лазерное оружие**: научно-фантастический лазерный пистолет с механикой нагрева/остывания.
- **Визуальные эффекты**: перманентные обугленные следы на стенах (технология Decal/Multiply).
- **Компонентная архитектура**: HUD, AI и управление разбиты на независимые модули, общающиеся через глобальный `EventBus`.

## 🗺️ Система координат уровня Hotel Siberia

### Мировые оси

| Ось | `+` | `−` | Роль на уровне |
|-----|-----|-----|----------------|
| **X** | Восток | Запад | Поперёк коридора / глубина комнат |
| **Y** | Вверх | Вниз | Высота |
| **Z** | Юг | **Север (вглубь)** | Длина коридора |

> Уровень вытянут по **−Z**. `(0, 0, 0)` — вход в коридор у лобби. Коридор тянется от Z=0 до Z=−85.

### Высотная шкала

| Точка | Y | Описание |
|-------|---|----------|
| Центр пола (CSGBox) | `−0.25` | Пол толщиной 0.5 м |
| **Игровой пол** | `0.0` | Поверхность передвижения |
| Центр стен | `2.0` | CSGBox стен высотой 4 м |
| Нижняя грань потолка | `4.0` | |
| Центр потолка (CSGBox) | `4.25` | |
| Петля двери | `0.0` | Всегда на игровом полу |
| Дверной проём (высота) | `0.0 .. 2.5` | Полотно двери 2.5 м |

### Поперечное деление по X

```
 X=-13.7       X=-3.6  X=-3.5  X=0  X=+3.5  X=+3.6         X=+11.3
    ┌─────────────┤ wall ├──────────────┤ wall ├──────────────┐
    │ Двойные ном.│      │  К О Р И Д О Р  │      │ Одиночные │
    │ (10.2×12 м) │      │  X: ±3.6 (пол) │      │ (7.8×7.2м)│
    │ origin=-8.3 │      │  X: ±3.5 (стен)│      │ origin=7.1│
    └─────────────┤      ├──────────────┤      ├──────────────┘
                 WallE(1.2м)             WallW(1.2м)
```

> Пол коридора шире стен: `corridor_width + 0.2 = 7.2 м` (перекрывает
> обе комнатные зоны на 0.1 м с каждой стороны для устранения physics seam CSG).

### Координаты номеров (Data-Driven Layout)

Ранее мы использовали алгоритмические формулы ($Z_L(n) = 4.0 - (n-1)*12.0$). 
В актуальной версии генератора (см. `hotel_level_generator_geometry.gd`) используется **data-driven подход**. Координаты Z для каждой комнаты прописаны явно через массивы словарей `left_rooms_data` и `right_rooms_data`.
Это позволяет точно соблюдать реальные чертежи отеля (например, разрыв на лифтовой холл 16м на левой стороне, отсутствие четкой симметрии на этаже, разные зазоры между номерами для технических помещений).

**Пример размещения левой стороны (DoubleRooms):**
- 408: `Z = 5.0`
- 406: `Z = -5.0`
- 405: `Z = -15.0`
- 403: `Z = -25.0`
- 402: `Z = -41.0` (обратите внимание на разрыв — это Лифтовой холл и Подсобка!)
- 401: `Z = -51.0`

**Пример размещения правой стороны (SingleRooms):**
- 421: `Z = 7.0`
- 420: `Z = 1.0`
- 417: `Z = -9.0` (и т.д., включая разрывы для лестниц и вентиляционных шахт)

> **Генерация стен:** Двери комнат больше не вырезаются с помощью `CSGHole3D`. Вместо этого коридорные стены генерируются отдельными секциями (`CorrWall_L_0`, `CorrWall_L_1`), оставляя точные физические промежутки шириной `room_door_opening_width` (1.2м) там, где расположены двери комнат и лестниц. Это решает проблему невидимых "physics seams" для CSG и значительно улучшает производительность на мобильных устройствах.

### Дверные проёмы в стенах коридора

Центр проёма по Z = **Z_центр_комнаты + 0.5** (для обоих типов комнат).
Зазор в стене коридора: **±1.0 м** от центра проёма.

| Дверь | Стена | Плоскость двери | Размер проёма (Ш×В) |
|-------|-------|-----------------|----------------------|
| DoubleRoom MainDoor | WallE, X=4.0 | YZ | 1.2 × 2.5 м |
| DoubleRoom WCDoor | WC_WallW, X=1.75 | YZ | 0.8 × 2.5 м |
| SingleRoom MainDoor | WallW, X=−3.0 | YZ | 1.2 × 2.5 м |
| SingleRoom WCDoor | WC_WallS, Z=−0.25 | XY | 0.8 × 2.5 м |

### Объект `entities/props/door.tscn`

```
DoorRoot  (Node3D)  ← позиция инстанса (шарнир петли, Y=0.0)
├── AnimatableBody3D  ← скрипт door.gd; ось вращения = происхождение тела
│   ├── MeshInstance3D   offset=(0.75, 1.25, 0)  BoxMesh 1.5×2.5×0.1 м
│   └── CollisionShape3D offset=(0.75, 1.25, 0)  BoxShape3D 1.5×2.5×0.1 м
├── SfxOpen   (AudioStreamPlayer3D)
└── SfxClose  (AudioStreamPlayer3D)
```

**Система координат двери:**
- `AnimatableBody3D.basis.z` — всегда указывает **В КОРИДОР** (наружу из комнаты)
- `+basis.z * N` → позиция в коридоре; `-basis.z * N` → позиция внутри комнаты
- Панель двери (1.5 м) крепится от шарнира в сторону `+basis.x`

**Логика открывания (`door.gd`):**
```gdscript
# interact() вызван от игрока из коридора → дверь открывается в коридор
open_angle = -PI/2   # при to_player.dot(basis.z) > 0
# interact() вызван от игрока изнутри → дверь открывается в комнату (рекомендуется)
open_angle = +PI/2   # при to_player.dot(basis.z) < 0
```

**Правило ориентации:**
- Коридорные двери (⊥ **X**): DoorRoot повёрнут **±90° вокруг Y**
- Санузел SingleRoom (⊥ **Z**): DoorRoot без поворота
- Шарнир всегда у **ближнего края** проёма, `Y = 0.0`

### Метки номеров (Label3D)

| Сторона | X позиция | Y | Формула Z |
|---------|-----------|---|-----------|
| Левые 401–408 | `−3.49` | `2.2` | `Z_L(n) + 0.75` (в центре проёма) |
| Правые 410–423 | `+3.49` | `2.2` | `Z_R(n) + 0.45` (в центре проёма) |

## 📝 Changelog

### 2026-07-13 — Фикс геометрии лестниц и Headless CSG багов

#### Архитектурная проблема наложения этажей
> **Проблема:** При генерации этажей `StairsSouthWall` 2-го этажа полностью перекрывал верхнюю половину (`StairsSouthWallTop`) 1-го этажа, наглухо блокируя верхнюю выходную дверь лестницы (`DoorHoleWest`) для игрока, пытающегося подняться.
> **Решение:** Понимание этой коллизии позволяет в будущем либо убрать перекрывающиеся стены, либо добавить симметричные проемы (`DoorHoleWestLower`) для сквозного прохода по петле этажей.

#### Баги `godot --headless` и CSG
> **Проблема:** В headless-режиме (на Jenkins) Godot использует `RendererDummy`, из-за чего откладывается или пропускается расчет булевых операций CSG. Проемы не вырезались, и автотесты врезались в глухие стены. Кроме того, метод `intersect_point` выдавал ложные срабатывания (ошибки "внутри/снаружи") на `ConcavePolygonShape3D`, сгенерированном из CSG.
> **Исправление автотестов:** 
> 1. В автотесты добавлена функция `_force_csg_update()`, принудительно вызывающая `node.get_meshes()` у каждого `CSGShape3D`. Это заставляет движок синхронно просчитать всю CSG-геометрию без рендерера.
> 2. Метод проверки `intersect_point` заменен на `intersect_ray` — физический луч, проходящий сквозь дверной проем, что гарантирует 100% точность (луч либо упирается в полигон стены, либо проходит сквозь "дырку").

### 2026-07-05 — Физические лестничные пролеты и многоэтажная вертикальная шахта

#### Физическая геометрия `north_stairs.tscn` и вырез в глобальном полу
> **Добавлено:** Полностью физически смоделирована П-образная лестница в блоке `north_stairs.tscn` (вместо старого одноуровневого заглушки). Она состоит из трех наклонных маршей (`EastFlight`, `NorthFlight`, `WestFlight`) и двух площадок (`NE Landing`, `NW Landing`).
> **Физика:** Лестница поднимается с уровня `Y=0.0` (восточная дверь этажа N) до уровня `Y=4.0` (западная дверь, выход к этажу N+1). Марши строго отцентрированы и сужены до ширины дверных проемов (1.2м) для точного стыкования без швов.
> **Вертикальная шахта (Генератор):** Скрипт `hotel_level_generator.gd` больше не рисует сплошной глобальный пол и потолок. Теперь он разбивает их на 3 куска (`South Main`, `North West`, `North East`), чтобы вырезать идеальное сквозное отверстие `7.2 x 4.8 м` прямо в зоне `north_stairs`. Это создает единый бесшовный вертикальный колодец между сгенерированными этажами 3, 4 и 5.

### 2026-07-05 — Фикс ориентации лифтовой шахты и дверей

> **Двери номеров:** Временно отключена генерация дверей (RoomDoor, WCDoor) в префабах номеров по просьбе игрока. Оставлена только дверь лифта.

### 2026-07-04 — Рефакторинг генератора уровней

#### Архитектура генератора (Hotel Level Generator)
> **Рефакторинг:** Массивный скрипт `hotel_level_generator.gd` разделен на три уровня для улучшения читаемости и поддержки, без потери настроек в инспекторе сцен:
> 1. **Base** (`hotel_level_generator_base.gd`) — Хранит `@export` переменные и базовые функции генерации фигур.
> 2. **Geometry** (`hotel_level_generator_geometry.gd`) — Логика расчета и построения геометрии коридоров и комнат.

### 2026-07-01 — Координатная сетка, physics seam и автотесты

> **Исправление:** `CorridorFloor` и `CorridorCeiling` расширены на 0.2 м (`corridor_width + 0.2 = 7.2 м`), создавая 0.1-метровое перекрытие в каждую комнатную зону.

> **Проблема:** Пол `SingleRoom` (размер X=8.4, центр X=0) имел левый край на global X=+2.9, не достигая границы коридора X=+3.5. Потенциальный physics seam.
> **Исправление:** `Floor` перемещён: центр X=+0.3, размер X=7.8. Левый край теперь на local X=−3.6 = global X=+3.5 (точно по границе коридора).
